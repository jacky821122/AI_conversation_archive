資料怎麼分層？哪層篩掉了什麼

五層轉換裡，真正大量去雜訊的只有 parser 那一層；jsonl→db 之間還悄悄少一個欄位、多兩個東西、改一次結構。下面的管線會收窄就是在表達這件事。

BEFORE title: "Used Gemini Apps"　（非 prompt 活動） AFTER（保留的長這樣） title: "Prompted 幫我寫…" → user_text = "幫我寫…"
ChatGPT附件只留指標，metadata 與本體都不留
image part 取 asset_pointer 指標字串記為 attachment，其餘 metadata（content_type / width / height / size_bytes / metadata{}）丟掉。圖檔本體（file-XXXX.dat）從頭到尾沒被讀——匯出夾用 conversation_asset_file_names.json 對照得回去，但 parser 沒接這條線。
BEFORE — JSON part（注意：裡面沒有位元組） { asset_pointer:"sediment://file_0000…", content_type, width, height, size_bytes, metadata{…} } ＋ 圖檔本體在匯出夾另存 file-XXXX.dat（此 export 共約 83MB）　← parser 從不讀取 AFTER attachments: ["sediment://file_0000…"]　← 只剩指標；到 db 層連這個也被丟

② parser → jsonl　不篩

write_jsonl 純序列化——記憶體有什麼就寫什麼，Gemini 在這裡仍是未縫合的 raw fragment。jsonl 是整條管線「資訊最完整」的一層。

③ jsonl → db　改結構（常被忽略）

−
attachments 欄位被丟掉
db schema 根本沒這欄
+
tokens
估算後寫入，供時間軸
+
FTS5 trigram
建全文檢索索引
⤙
Gemini 縫合
fragment 併成 session，對話變少

④ db → cli / web　內容一致

web 與 cli 共用 store.py 同一組函式，資料完全一樣。差別只在呈現層：search snippet 截斷（120/140 字）是顯示截斷，底層 text 仍是全的。

一句話總結

交界內容變化
raw → parser絕大多數過濾：非 prose、citation、廢 session、附件本體
parser → jsonl無（原樣落地，Gemini 仍是 raw fragment）
jsonl → dbGemini 縫合 · attachments 丟失 · +tokens · +FTS
db → cli/web內容一致，只有顯示截斷

⚠️ 唯一的「資訊靜默流失」

attachments 在進 db 時被丟掉——jsonl 還看得到附件指標，但你透過 cli/web 查不到。日後若想做「找出有附圖的對話」，得回頭把這欄帶進 db schema。
